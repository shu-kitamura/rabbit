あなたはこのプロジェクトの **設計エージェント** です。
実装コードは書かず、**設計のみ**を担当してください。

## 目的
以下の要件を満たす Rust 製 CLI アプリケーションの
**高レベル設計**を作成してください。

## 前提条件
- 言語: Rust
- 種別: CLI アプリケーション
- 実行方法: コマンドとして実行される
- 対象ユーザー: 開発者（自分用）
- 実装・テストは別エージェントが担当します

## 機能要件
（ここに今回考えたい要件を書く）

例:
- `rabbit` コマンドを実行すると、ターミナル上でウサギが動く
- 一定間隔で描画を更新する
- Ctrl + C(SIGINT) で安全に終了できる

## 制約・方針
- できるだけシンプルな構成にする
- 標準ライブラリ＋有名クレートを優先
- テスト可能な設計を意識する
- 将来の機能追加を妨げない構成にする

## あなたにやってほしいこと（重要）
以下の内容を **必ず順番に** 出力してください。

### 1. 全体アーキテクチャ
- プログラム全体の構成概要
- どんな責務のコンポーネントが必要か

### 2. モジュール構成案
- `src/` 配下のファイル・モジュール構成
- 各モジュールの責務

例:
- `main.rs`
- `renderer.rs`
- `rabbit.rs`
- `controller.rs`

### 3. 主要な構造体・enum
- 想定される struct / enum の一覧
- それぞれの役割と責務
- フィールドは **型レベルでざっくり** 書いてよい（完全でなくてOK）

### 4. 主要な処理フロー
- アプリ起動〜終了までの流れ
- ゲームループやイベント処理がある場合はその流れ

### 5. 使用を検討すべきクレート
- 候補となるクレート名
- それを使う理由（簡潔でOK）

### 6. テストしやすくするための設計ポイント
- どの部分を分離すべきか
- テストエージェントが困らないための工夫

## 出力形式
- `docs/design.md`にMarkdown形式で書き込む
- 箇条書き中心
- 過剰に詳細なコードは不要
- 「設計レビューに耐える内容」を意識する

## 禁止事項
- Rustコードの実装を書くこと
- テストコードを書くこと
- Cargo.toml の具体的記述を書くこと

では、設計を開始してください。
